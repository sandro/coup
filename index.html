coup

You need a template.
You need a component.
You need to decide when to render.

That's a coup.

The crime

Somewhere along the way, building a button that says "clicked 3 times" started requiring a bundler, a transpiler, a framework, a state management library, and 200MB of node_modules heavier than the application they exist to serve. Build pipelines now take longer to configure than the feature they enable.

Reactivity models got so complex that frameworks ship dedicated devtools to help you debug their own rendering. Think about that. The framework can't explain itself without a browser extension.

The frameworks created problems that didn't exist before them — stale closures, zombie children, render cascades, hydration mismatches — then sold us the solutions: useMemo, useCallback, React.memo, Suspense. An industry of problems that only exist inside the tool meant to solve them.

The lie

The central lie of modern frontend is: "you cannot be trusted with the render loop."

React, Vue, Svelte, Angular — they all decide when to render for you. They call it "declarative." They say it frees you to think about what instead of when.

But every performance issue you've ever debugged was a when problem. Why did this re-render? Why did the whole tree update when one input changed? Then you spend your afternoon wrapping things in memo and useCallback, manually telling the framework when not to render — which is just telling it when to render, with extra steps and a dependency array you have to maintain by hand.

The framework took away your control, then gave you back a worse version of it and called it an optimization API. Time to take it back.

The counter-offer

What if you just… called render() when you were ready?

import { CoupElement, html } from 'coup'

class TodoList extends CoupElement {
  static tag = 'todo-list'

  state = { items: [], text: '' }

  addItem() {
    if (!this.state.text.trim()) return
    this.state.items.push(this.state.text)
    this.state.text = ''
    this.render()
  }

  template() {
    return html`
      <input
        .value=${this.state.text}
        @input=${e => this.state.text = e.target.value}
      >
      <button @click=${() => this.addItem()}>Add</button>
      <ul>
        ${this.state.items.map(i => html`<li>${i}</li>`)}
      </ul>
    `
  }
}

TodoList.define()

No hidden machinery. No dependency arrays. No stale closures. No rules of hooks. You changed state, called render(), lit-html diffed the template, and the DOM updated. You just staged a coup against your own build pipeline.

What coup is

~250 lines of code. That's the entire framework — not just the "core," not the "runtime," not the part before the plugins load. The whole thing. You can read it in one sitting, understand every line, and never wonder what happens between your state change and the DOM update.

The "build system" is an <script type="importmap">: 

<script type="importmap">
{
  "imports": {
    "lit-html": "https://esm.sh/lit-html@3",
    "lit-html/": "https://esm.sh/lit-html@3/",
    "coup": "./index.js"
  }
}
</script>
<script type="module" src="app.js"></script>

No npm install. No webpack.config.js. No vite.config.ts. No next.config.mjs. Open the HTML file. It works.

Under the hood, coup uses custom elements — the browser's own component system. Shipped in every browser since 2020. Ignored by an industry addicted to compilation.

Lifecycle you can hold in your head

Five hooks. No dependency arrays. No cleanup-return-function guessing games.

  • connected() Element added to the DOM. Set up your state, start timers, fetch data.
  • propsChanged(changes) One or more props changed. You get an object with { propName: { old, new } }.
  • storeChanged(store, state) A subscribed Store updated. You decide if and when to render.
  • updated() DOM just re-rendered. Measure elements, init third-party widgets, scroll.
  • disconnected() Element removed. Clean up whatever you started.

Compare with React's useEffect — a single hook that handles mounting, updating, and cleanup, distinguished only by what you put in the dependency array. Miss a dependency? Stale closure. Include too many? Infinite loop. The old class component lifecycle had componentDidMount, componentDidUpdate, componentWillUnmount, getDerivedStateFromProps, shouldComponentUpdate, getSnapshotBeforeUpdate… a carousel of methods nobody memorized.

Coup's lifecycle is five methods that do what their names say. No surprises.

The proof

Seven runnable examples. No installs. No builds. Click and read the source. Every one is a single HTML file with an importmap and vanilla JavaScript.

  • 1. Hello World
    2 components, prop passing, input binding
  • 2. Task List
    Add, remove, reorder — keyed lists with repeat()
  • 3. Kanban Board
    Drag-and-drop columns, CRUD, drop preview
  • 4. Chat Rooms
    Room switching, component destruction, timers
  • 5. GitHub Explorer
    Fetch API, async on mount, loading/error states
  • 6. Bookmarks
    Shared Store, search, tag filtering, derived data
  • 7. Block Editor
    Block editor — Tiptap + CodeMirror, bidirectional sync

Get started

Every .html file you open without a build step is a small act of rebellion. Start here:

git clone https://github.com/sandro/coup.git
cd coup
open examples/1-hello/index.html

No install. No build. No wait. That's a coup.
