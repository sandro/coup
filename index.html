coup

A coup against the tyranny of the modern frontend.

The crime

Somewhere along the way, adding a little interactivity to a web page started requiring a bundler, a transpiler, a framework, a state management library, and 200MB of node_modules heavier than the application they exist to serve. Build pipelines now take longer to configure than the feature they enable.

Reactivity models got so complex that frameworks ship dedicated devtools to help you debug their own rendering. Think about that. The framework can't explain itself without a browser extension.

The frameworks created problems that didn't exist before them — stale closures, zombie children, render cascades, hydration mismatches — then sold us the solutions: useMemo, useCallback, React.memo, Suspense. An industry of problems that only exist inside the tool meant to solve them.

The scene of the crime

Ask a React developer what happens when they call setState. Not what they think happens — what actually happens. Which components re-render? In what order? Why did that child re-render when its props didn't change?

They don't know. Most of them have never known. They learned the API, not the machine. And the machine was designed to be invisible — until it breaks, and then you're debugging a 15-frame stack trace through a reconciler you were never meant to understand.

React's answer is more API. useMemo to stop computing things you shouldn't be computing. useCallback to stabilize references that are only unstable because React tears down and re-creates your function on every render. React.memo to prevent re-renders that only happen because the parent re-rendered for unrelated reasons. Each "optimization" is a patch for a wound the framework inflicted.

And they keep adding. Server Components. use(). The compiler that will finally make useMemo unnecessary — which is an admission that they shipped a broken performance model and asked developers to fix it by hand for eight years.

What's actually hard

Frontend development is not hard. React is hard.

Components, templates, state, events — these are simple ideas. A component is a class. State is a property. A template is a function that returns markup. An event is a function call. Every developer already understands these concepts. Frameworks took simple concepts and buried them under layers of abstraction until "senior frontend engineer" became a specialization in navigating framework internals rather than building interfaces.

The browser has had a native component model since 2020. Custom elements. Observed attributes. Lifecycle callbacks. The platform caught up years ago. The industry didn't notice because it was too busy configuring Webpack.

The counter-offer

Drop a script tag into any page. Add components where you need them. No rewrite. No migration. No build step. 

import { CoupElement, html } from 'coup'

class MyCounter extends CoupElement {
  static tag = 'my-counter'
  state = { count: 0 }

  template() {
    return html`
      <button @click=${() => { this.state.count++; this.render() }}>
        Clicked ${this.state.count} times
      </button>
    `
  }
}

MyCounter.define()

That's a real component. You own every render. Efficient DOM updates, automatic cleanup. It works in a static HTML page, a Rails app, a WordPress theme, a marketing site — anywhere you can put a <script> tag.

Need a search box with autocomplete? A form with validation? A live-updating dashboard? Drop in one component. The rest of your page doesn't know or care that coup exists.

Need a full app? Keep going. Parent-child props, shared stores, event-driven communication between components — coup scales from a single widget to an entire SPA. Same tool, same patterns, no second framework when things get "serious."

The honest part

Coup doesn't auto-render when you change state. You call this.render(). Every other framework hides this from you — signals, proxies, dirty-checking, compiler magic. Coup doesn't. You mutate your state, you tell the DOM to update. That's it.

This is a feature. When the UI is wrong, you grep for missing render() calls. That's an obvious, findable bug. When a framework auto-renders, the failure mode is invisible over-rendering — components re-rendering 30 times a second because a function reference changed, or an event handler fired more often than you expected. Good luck grepping for that.

What coup offers is a small surface area. Under 300 lines. One file. One dependency (lit-html, ~3KB). No compiler. No virtual DOM. No reconciler. No fiber architecture. No hydration. No server runtime. No CLI. No starter template.

You can read the entire framework during a coffee break. Not skim it — read it, understand every line, and know exactly what happens between your state change and the DOM update. No hidden machinery. No trust required.

The coup is not novelty. The coup is that you don't need 50,000 lines of framework to build for the web — whether that's one component on an existing page or an entire application from scratch. The browser has been ready for years. The industry just kept building on top of it instead of using it.

What coup actually fights

The rewrite. You don't adopt coup. You drop it in. Add a component to one page, see if you like it, add another. Or build the whole thing from scratch. Either way, coup components are custom elements — the browser already knows what to do with them.

The build step. Your counter doesn't need Webpack. Your todo list doesn't need Vite. An import map and a script tag — that's the entire "build system." Open the HTML file. It works. Ship it.

The invisible machinery. No virtual DOM diffing your entire tree. No reconciler deciding which children to re-render. No fiber scheduler slicing your work into chunks you didn't ask for. When you call this.render(), lit-html patches only the expressions that changed. That's it. You could step through it in a debugger in under a minute.

The invented problems. No stale closures — your methods are methods, this is this. No dependency arrays — lifecycle hooks fire when their names say they fire. No rules of hooks — there are no hooks. State is a property. A handler is a method. An effect is code you write in connected() and clean up in disconnected().

The expertise tax. You don't need to be a "React developer" to use coup. You need to know JavaScript and HTML. That's it. The component model is a class. The template is a tagged template literal. The state is an object. If you can write a class with methods, you can build anything coup builds.

Components you can trust

When you need a component, you should be able to reach for it without hesitating. Will it over-render? Will it be hard to debug? Will it add overhead? With coup, the answer is always no — and you don't have to take our word for it, because you can see it.

It can't over-render. A coup component renders when you call this.render() or when a parent changes its props. That's it. No tree reconciliation. No cascading re-renders from an ancestor. No "why did this component re-render?" mystery. Zero renders happen unless someone explicitly asked for one.

It won't be hard to debug. Search your code for this.render(). Every render is a line you wrote. Step through in a debugger — you'll see your state change, lit-html diffing your template expressions, and the DOM update. Five stack frames, not fifty. The entire path from state change to pixel is visible without framework-specific tooling.

It won't add overhead. A coup component is a custom element — the same thing the browser uses for <video> and <input>. No wrapper divs, no synthetic event system, no runtime scheduler. The component is the DOM node. Drop it into any page and the cost is the code you wrote, nothing more.

What we learned

Coup originally shipped with this.render() as the only way to update the DOM. Then we added reactive state — static state = { count: 0 } — because it seemed simpler. Set a property, the component re-renders. No boilerplate. Elegant. We were very proud of ourselves.

Then we built a real app with it. A Spotify playlist manager with external SDK events, streaming callbacks, timers, and child components. And we found the bugs.

Bug 1: .bind() in templates creates a new function object every render. The child sees a new prop, re-renders. A component re-rendering once per second (playback timer) caused its children to re-render once per second too — for no reason.

Bug 2: An SDK firing player_state_changed five times per track change. Each event set the same state to an equal-but-not-identical object. Five renders where zero were needed. And we didn't notice until we added debug logging.

These are exactly the problems coup was built to prevent. We wrote a framework to stop invisible over-rendering, then added a feature that reintroduced it. Even we fell for the reactivity trap.

So we reverted. Removed reactive state entirely. Went back to this.render(). Then we added a two-line guard — if the state didn't actually change, don't render. 169 renders per play click → 1. Zero renders during 10 seconds of playback. That guard was trivial to write because this.render() is a line you choose to add. Under reactive state, preventing a render meant understanding the framework's equality semantics. Much harder problem.

The full story: Reactive State Postmortem

Lifecycle you can hold in your head

Six hooks. No dependency arrays. No cleanup-return-function guessing games.

  • connected() Element added to the DOM. Set up your state, start timers, fetch data.
  • firstUpdated() Fires once after the first render. DOM is populated — safe to query, measure, focus.
  • updated() DOM just re-rendered. Measure elements, init third-party widgets, scroll.
  • disconnected() Element removed. Clean up whatever you started.
  • propsChanged(changes) One or more props changed. You get { propName: { old, new } }.

Compare with React's useEffect — a single hook that handles mounting, updating, and cleanup, distinguished only by what you put in the dependency array. Miss a dependency? Stale closure. Include too many? Infinite loop. connected runs on mount. disconnected runs on unmount. That's the whole story. No return functions. No gotchas.

Get started

I use coup both ways — drop-in components on existing pages and full single-page apps built entirely with it. Same tool. Same patterns. I hope you will too.

Every .html file you open without a build step is a small act of rebellion. Start here:

git clone https://github.com/sandro/coup.git
cd coup
open examples/1-hello/index.html

No install. No build. No wait. That's a coup.
